# Distributed Provenance Demo

This demo presents application of the Common Provenance Framework (CPF/ISO 23494) on a use case from Digital Pathology
at Marasyk Memorial Cance Institute (MMCI, Czech Republic) and RationAI group (Masaryk University, Czech Republic).
The goal of the demo is to evaluate the Frameworks ability to support trustworthy and
confidentiality-preserving traceability of information in multi-organizational environments using provenance information. This demo
serves as the implementation part for the publication available at [Zenodo](https://doi.org/10.5281/zenodo.14526107). All details about the
framework, tested hypotheses and their evaluation can be found there.

In this demo, three parties are simulated - a surgery department, a pathology department, an AI research group.

Each of the parties is represented in the demo by 3 docker instances:
- A docker instance representing the legacy systems (the Hospital Information System or MLFlow), based on which the standardized provenance information (finalized provenance component in terms of the Common Provenance Model) is generated. The docker instance for the Surgery and for the Pathology is shared, as there is a shared information system also in the production environment.
- A docker instance containing the Provenance Storage System. The System enables the storage of standardizef finalized provenance components.
- A docker instance containing a database for the Provenance Storage System.

In addition, two docker instances represent a Trusted Party, which supports non-repudiation of origin of finalized provenance components.
These intances contain Trusted Party Service and its database respectively. The queries implemented in this demo are executed over the Provenance Managenent System instances.

## Getting Started

### Dependencies

* Docker • [Install](https://docs.docker.com/engine/install/)

### Executing provenance generation

Clone this repository
```shell
git clone git@gitlab.ics.muni.cz:422328/dbprov.git
```
Enter the project directory
```shell
cd dbprov
```
Build docker images
```shell
docker build -f Dockerfile.Mlflow -t dbprov-mlflow .
docker build -f Dockerfile.Provenance -t dbprov-provenance .
docker build -f Dockerfile.ProvStorage -t dbprov-prov-storage .
docker build -f Dockerfile.TrustedParty -t dbprov-trusted-party .
```
Start the demo
```shell
docker compose up -d
```
Wait for all the containers to start - this may take about 2 minutes. What happends during the compose:

- Dockerfile.Provenance -- initialize data for the Surgery and Pathology departments. Patients data in a simulated database are generated synthetically. This synthetic data corresponds to the data used by the Masaryk Memorial Cancer Institute. The image data (WSI's) generated by the Pathology department are represented by the CAMELYON16 dataset. This dataset is pre-prepared in the data/CAMELYON16 directory, which is mounted to the docker image.
- Dockerfile.Mlflow -- initialize data for the AI Group. The data that are imported into the MLFLow instance come from the RationAI groups MLFlow instance and were modified manually so that the filenames used in the MLFlow isntance correspond to the CAMELYON16 dataset. Only the filenames are changes, the structure of the dataset is the same.
- Dockerfile.ProvStorage -- initialize finalized provenance components for all three parties (implementation of the finalization event). Python scripts convert the information from the source systems to the standardized representation which is stored here in the Provenance Managemet System instances.
- Dockerfile.TrustedParty -- initialize the Trusted Party for the provenance chain. Initiating scripts registers the parties in the system, so that the Provenance Management System can use the service.

## Technical notes

### Provenance Storage & Trusted Party Servers

The implementation of the Provenance Storage System and the Trusted Party service was initially developed by David Gajdoš [here](https://github.com/GajdosDavid/distributed-provenance-system). The exposed REST API endpoints can be found here: [Provenance Storage API](https://app.swaggerhub.com/apis/DAVIDGAJDOS/provenance_storage/1.0.0) and [Trusted Party API](https://app.swaggerhub.com/apis/DAVIDGAJDOS/trusted_party/1.0.0#/). Both services were later tested and refactored by Martin Mojžiš [updated code available here](https://is.muni.cz/th/du7ec/).

We have took over the Provenance Storage System implementation and further developed it so that it reflects the changes that were made to the CPM (e.g., the implementation now supports specialized forward connectors , fixed `cpm:id` entities are part of domain-specific subgraph, etc.).The trusted party server has been used as is without modifications.

### Simulation & Provenance Generation

The simulation of hospital and pathology workflow is performed in the `populate_db.py` Python script. Based on the parameters supplied in the `compose.yaml` file, the predefined number of patients and staff members are generated. All names, people and hospital, are randomly generated from [a list of names](https://github.com/dominictarr/random-name/blob/master/) found on Dominic Tarr's github.

Additionally, the input WSI slides are divided among the generated patients based on the following rules:
* healthy patients can only be assigned healthy WSIs (i.e. `training/normal/normal_XXX.tif`)
* each healthy patient receives at least `MIN_SLIDES_PER_HEALTHY_PATIENT` WSIs
* remaining healthy WSIs are randomly distributed between healthy and tumour patients using a binomial distribution with $p=$ `OVERFLOW_PERCENTAGE` being the probability of being assigned to a healthy patien.
* tumour patients can be assigned any WSIs with at least one being a tumour WSI (i.e. `training/tumor/tumor_XXX.tif`)
* test patients are assigned test WSIs (i.e. `testing/images/test_XXX.tif`)
* for each group of patients (normal/tumor/test) `REPEAT_PERCENTAGE`% patients undergo more than one biopsy

The records about patients, staff, surgeries and tissue analyses are stored in the Postgres database using the [Pony ORM](https://docs.ponyorm.org/) package. The `entity.py` describes database entities/tables and relationships between them:
* **Hospital**:
  * hospital name
  * scanner vendor
  * list of surgeons
* **Patient**:
  * random patient UUID
  * full name
  * consent
  * patient group (healthy/tumour/test)
  * list of provided tissue samples
* **Employee**:
  * random employee UUID
  * full name
  * position (surgeon/pathologist)
  * list of samples (if surgeon) or analyses (if pathologist) they were responsible for
  * organization
* **Sample**:
  * random tissue sample UUID
  * patient
  * surgeon
  * timestamp of surgery
  * analysis
  * provenance URI
  * meta provenance URI
* **Analysis**:
  * random analysis UUID
  * pathologist
  * diagnosis (tumor/normal)
  * block number
  * timestamp of analysis
  * WSI URI
  * provenance
  * meta provenance URI

Prior to provenance generation, each party (surgery, pathology, AI group) is registered with their respective Provenance Storage System in the Trusted Party. To do so, a private key and a certificate is generated for each party. We are following the certificate scheme:

> CA -> Intermediate Cert 1 -> Intermediate Cert 2 -> Organization Cert

The Trusted Party trusts the root CA certificate and any certificate whose chain of trusts originates at the root CA certificate. All organization certificates are generated using the second Intermediate Cert. If the chain of trust cannot be traced to the root CA certificate, any attempt to store new or modify existing provenance documents will fail.

Finally, the provenance is generated using hand-crafted templates for domain-specific parts of the provenance. The surgery and pathology provenance is generated using database records created during simulation, while research provenance is generated using MLflow experiment runs. The MLflow experiment runs were imported from RationAI MLflow instance using [mlflow-export-import](https://github.com/mlflow/mlflow-export-import/#egg=mlflow-export-import) package.

### Databases

Databases docker images are run as-is. The only modification is the mandatory change of the default password.
